Overview

Siesta is an object relational mapper for php build for performant database access. It allows to define objects and there mapping to tables in a database. Objects can be intialized with http request data or with json data. That allows to handle JavaScript objects in the browser, send them to the backend, parse the data, manipulate the data and then persist it. Currently only drivers for for MySQL are available.

Important features:

Code Generation
Instead of writing a large generic engine with tons of code, the needed code for persistency operations is generated from descriptions. Therefore the needed persistency footprint is very small, hence fast.

Stored Procedures insted of ad-hoc SQL
Siesta relies to 100% on Stored Procedures. The generator creates all needed stored procedures directly in the database. In contrast to classic ad hoc sql the stored procedures are precompiled in the database which increases performance. Furthermore custom stored procedures can be defined.

Extensibility
Code generation approaches tend to limit programmers ability to change behaviour. Therefore an import design principle is to allow for extensibilty at every given point.

In Memory replication
Siesta for MySQL allows to replicate tables with a memory engine. That means the table exist twice once with the defined engine and one with the memory engine. All modifying operations are executed on both tables. Read operations are only performed on the memory table.

Delimiting Tables / Record History
Sometimes it is important to track changes to objects. Therefore Siesta allows to generate a second history table. Every time the a record is changed, the changes will be stored to the history table with a valid from and valid until entry.

Hello World

The example is available in the distribution. Just run the commands to generate the classes. They will be created in the folder helloworld folder. 

    vendor/bin/siesta init
    vendor/bin/siesta generate --entityFileSuffix .hello.xml

The following code shows how the classes can be used. First Artist and Label object are instantiated. Then the relation between both is established by $label->addToArtistList($artist);. The save method can be invoked with true that will result in a cascading save. That means that both objects will be stored. 

<?php
    namespace test;

    require_once 'vendor/autoload.php';
    require_once ('helloworld/Artist.php');
    require_once ('helloworld/Label.php');

    use helloworld\Artist;
    use helloworld\Label;

    $artist = new Artist();
    $artist->setName("Jamie Woon");

    $label = new Label();
    $label->setName("PMR");
    $label->setCity("London");
    $label->addToArtistList($artist);

    // save with cascade. will store both label and artist
    $label->save(true);

The XML used describes 2 entities. One named Artist and one named Label. Both entities have simple attributes with the given php and database types. Furthermore one relation is defined. The Artist <reference> element describes a relationship to the Label class. The Label class has a <collector> element that describes the foreign class and the name of the reference. The XML can be found in vendor/gm314/siesta/HelloWorld.hello.xml. 

    <entityList>

        <entity name="Artist" namespace="helloworld" targetPath="helloworld">
            <attribute name="id" type="int" dbType="INT" primaryKey="true" autoValue="autoincrement"/>
            <attribute name="name" type="string" dbType="VARCHAR(100)"/>

            <reference name="label" foreignClass="Label" required="false"/>
        </entity>

        <entity name="Label" namespace="helloworld" targetPath="helloworld">
            <attribute name="id" type="int" dbType="INT" primaryKey="true" autoValue="autoincrement"/>
            <attribute name="name" type="string" dbName="NAME" dbType="VARCHAR(100)"/>
            <attribute name="city" type="string" dbName="CITY" dbType="VARCHAR(100)"/>

            <collector name="artistList" type="1n" foreignClass="Artist" referenceName="label"/>
        </entity>

    </entityList>

Console Commands

Init

First you shoud use vendor/bin/siesta init to create a config file. It will ask you about the connection details.


Generate

The command vendor/bin/siesta generate allows to generate the classes, tables and stored procedures for your entity files. Per default it will search in the current working directory for all files that end with .entity.xml. If you set an option on the command line, it will overwrite the configured value in the siesta.config.json. Available options are:

Option Default Description
--configFile you can specify a config file to use. If not supplied siesta will search for the first siesta.config.json file it finds
--dropUnusedTables false add this option to have siesta drop tables that are not defined anymore
--baseDir current working directory define where siesta should look for entity files and where to store generated classes
--connectionName will use the connection that has the default flag set to true If you have configured several connections in the siesta.config.json file, you can select a connection name here
--entityFileSuffix .entity.xml siesta will search for files in the baseDir that end with the given entity file suffix.
--migrationMethod direct With this option you define if tables are altered directly (direct) or if the ALTER statement are written to a file
--migrationTargetPath . Define where the migrations scripts should be stored to.

Reverse Engineering

The command vendor/bin/siesta reverse will create entity files for the schema found in the database. You have the following options.

Option Default Description
--configFile you can specify a config file to use. If not supplied siesta will search for the first siesta.config.json file it finds
--connectionName will use the connection that has the default flag set to true If you have configured several connections in the siesta.config.json file, you can select a connection name here
--entityFileSuffix .entity.xml siesta will name the generated xml files {TableName}{entityFileSuffix}
--targetPath reverse define where you want to store the generated xml files
--targetNamespace "" Define which namespace should be used
